== Executive Summary
This document captures the end-to-end secure boot and storage architecture implemented on the i.MX93 platform. It explains the chain of trust, runtime protections, authenticated storage design (dm‑integrity + dm‑crypt), threat mapping (STRIDE), and compliance mappings (ISO/SAE 21434 and UNECE R155).

The design now includes **dm‑integrity with HMAC‑SHA256**, providing *authenticated integrity* for the writable `/data` filesystem, in addition to dm‑crypt AES‑XTS encryption and OP‑TEE–backed key protection.

== Architecture Overview
* *AHAB Secure Boot (ROM)* verifies the signed boot container (`flash.bin`) containing TF‑A (BL31), OP‑TEE, and U‑Boot.
* *U‑Boot* verifies a signed FIT image (kernel, DTB, initrd, boot script) using an embedded RSA‑2048 public key.
* *Linux kernel* boots with *dm-verity* enforcing an immutable, Merkle-verified root filesystem.
* *Initrd*:
  - Starts the OP‑TEE supplicant.
  - Unwraps two device-bound keys via OP‑TEE:
    **(1)** dm‑crypt AES‑XTS 512‑bit key  
    **(2)** dm‑integrity 32‑byte HMAC key (raw binary)
  - Opens *dm-integrity* on `/dev/mmcblk1p6`.
  - Opens *dm-crypt* on top of the integrity-mapper.
  - Mounts the resulting stack at `/data`.
* *overlayfs* provides writable persistence using:
  - `lowerdir=/` (verity-protected root)
  - `upperdir=/data/overlay/upper` (encrypted & integrity-protected)

== Runtime Protections
* *AHAB ROM* authenticates the boot container before executing any untrusted code.
* *FIT signatures* authenticate kernel, DTB, initrd, and the boot script.
* *dm-verity* enforces an immutable, integrity-verified root filesystem.
* *OP-TEE* unwraps keys using AES‑GCM with a KEK derived from hardware-bound secrets via system PTA.
  - The wrapped keys at rest cannot be decrypted or tampered with outside OP‑TEE.
* *dm-integrity* (HMAC-SHA256)
  - Uses per-sector authentication tags to detect tampering or corruption.
  - Guarantees atomic writes of data+tags via a journal.
* *dm-crypt* (AES‑XTS 512-bit)
  - Provides confidentiality for all user data at rest.
* *Authenticated encryption-at-rest*:
  - AES‑XTS for confidentiality  
  - HMAC-SHA256 for per-sector integrity  
  - AES‑GCM only for key wrapping  
* *overlayfs* ensures all writable system state is stored inside the encrypted and integrity-protected `/data`, keeping the base OS immutable.

== Authenticated Storage (dm-integrity + dm-crypt)
The writable `/data` partition is protected by a two-layer mapper:

1. **dm-integrity**  
   *Integrity provider*  
   * Key: 32‑byte raw HMAC key  
   * Algorithm: `hmac(sha256)`  
   * Tag size: 32 bytes  
   * Journal: ensures atomic writes  
   * Detects tampering, replay, and silent corruption.

2. **dm-crypt (AES-XTS)**  
   *Confidentiality provider*  
   * Key: 512‑bit XTS key (two 256‑bit keys)  
   * Encryption only—no integrity  

Together they form an *authenticated encryption-at-rest* system.

== STRIDE Threat Mapping
[cols="20,25,55", options="header"]
|===
| Threat | Surface/Asset | Mitigation

| Spoofing
| Boot components, keys
| AHAB secure boot; FIT signatures; OP‑TEE key unwrapping

| Tampering
| `/data`, dm-crypt keyblob, integrity tags
| dm-verity; dm-integrity HMAC; AES-GCM-wrapped keys; integrity journal

| Repudiation
| Provisioning, updates
| Signed artifacts; auditable build logs; persistent key counters in RPMB

| Information Disclosure
| Encrypted `/data`
| dm-crypt AES-XTS; OP‑TEE KEK; no plaintext on storage

| DoS
| Integrity metadata, storage
| dm-integrity journal; recovery; enforced atomic writes

| Elevation of Privilege
| Writable filesystem, kernel
| Immutable dm-verity root; authenticated upper layer; prevents persistent malware
|===

== Operational Notes
* Program OTP/SRK fuses to permanently enable secure boot.
* Lock debug (SJC) as policy allows.
* Keep FIT signing keys offline.
* Maintain reproducible builds and metadata for audit trails.
* Ensure initrd always includes: `integritysetup`, OP‑TEE clients, and TAs.

== Appendix A — Configuration Parameters
* FIT signatures: RSA‑2048, SHA‑256
* dm-verity:
  - Merkle tree over rootfs  
  - `roothash` + `salt` passed via kernel cmdline
* dm-crypt:
  - AES‑XTS with 512‑bit key  
  - XTS key unwrapped via OP‑TEE and hex-encoded into dm‑crypt table
* dm-integrity:
  - Algorithm: `hmac(sha256)`
  - Tag size: 32 bytes
  - Journal size: 32 MiB
  - Integrity key: **raw 32-byte file** (`/run/keys/integrity.key`)
* OP-TEE:
  - KEK derived via system PTA hardware secrets
  - AES‑GCM-wrapped key blobs stored in keyfs
  - RPMB counters protect key rollback
